# 03 - PRD（产品需求文档）

> Project: Smart Salary Engine（SSE）  
> Version: V1.2  
> Status: MVP 可开发基线

---

## 1. 背景与问题

HR 在工资核算时经常需要处理多份 Excel：工资主表、考勤表、奖金表、扣款表、社保公积金表、临时补充表等。不同文件的 Sheet 名、列名、顺序、格式不稳定，人工整理容易出现漏算、错算、覆盖来源不清、最终工资无法解释等问题。

SSE 的目标是把这套 Excel 工资核算过程变成可检查、可追溯、可解释的半自动流程。

---

## 2. MVP 范围

### 2.1 MVP 必做功能

| 编号 | 功能 | 说明 |
|---|---|---|
| F01 | 创建工资核算任务 | 以工资月份和任务名称管理一次核算 |
| F02 | 上传工资主工作簿 | 支持 `.xlsx` 文件上传和保存 |
| F03 | 上传补充 Excel | 支持多次导入补充数据 |
| F04 | Sheet 自动识别 | 根据 Sheet 名、表头关键词、数据结构识别类型 |
| F05 | 列名自动识别 | 将原始列映射到标准字段 |
| F06 | 人工确认识别结果 | Sheet/列识别低置信度时允许人工修正 |
| F07 | 员工数据池 | 按姓名聚合员工工资数据 |
| F08 | 重名处理 | 用户保证无重名；系统发现重名时 BLOCK，要求检查 Excel |
| F09 | 数据合并 | 多批次导入后补齐字段并发现冲突 |
| F10 | 数据检查 | 检查缺失、重复、逻辑、金额异常 |
| F11 | 异常处理 | BLOCK/WARN/INFO 分级处理 |
| F12 | 工资计算 | 按配置公式计算工资 |
| F13 | 计算过程保存 | 保存公式、参数、来源、结果 |
| F14 | 工资解释 | 按员工展示工资明细和计算依据 |
| F15 | Excel 导出 | 基于工资主表模板导出最终工资表、计算过程、异常报告 |
| F16 | 基础登录 | MVP 支持本机单用户简单登录 |

### 2.2 MVP 不做功能

- 员工档案维护；
- OA 审批流；
- 考勤设备直连；
- 社保/个税/银行接口；
- 个税计算；
- 多租户复杂权限；
- 手机端；
- 实时协同编辑。

## 3. 用户故事

### US01 创建工资核算任务

作为 HR，我希望按月份创建一个工资核算任务，方便把本月所有 Excel 导入、检查、计算、导出都归到同一个任务下。

验收标准：

- 可以输入任务名称、工资月份、备注；
- 创建成功后生成任务 ID；
- 任务初始状态为 `CREATED`；
- 任务列表可查看任务状态、创建时间、最后更新时间。

---

### US02 上传工资主工作簿

作为 HR，我希望上传工资主 Excel，让系统自动解析 Sheet 和列名。

验收标准：

- 支持 `.xlsx`；
- 单文件大小 MVP 默认不超过 50MB；
- 上传后生成导入批次；
- 原始文件保存，不被修改；
- 文件损坏时给出明确错误；
- 解析完成后展示 Sheet 识别结果。

---

### US03 Sheet 自动识别与人工确认

作为 HR，我希望系统自动识别哪些 Sheet 是工资主表、奖金表、扣款表，识别不准时我可以手动选择。

验收标准：

- 每个 Sheet 显示识别类型和置信度；
- 置信度低于阈值时标记“需确认”；
- 用户可以修改 Sheet 类型；
- 修改后保存到当前任务；
- 未确认的关键 Sheet 会产生 BLOCK 异常。

---

### US04 列名自动识别与人工映射

作为 HR，我希望系统自动把“基本薪资”“底薪”“基本工资”识别成同一个标准字段。

验收标准：

- 展示原始列名、标准字段、置信度；
- 低置信度列可人工选择标准字段；
- 必填字段未映射时产生 BLOCK 异常；
- 人工映射保存到当前任务；
- 可选择是否保存为全局规则。

---

### US05 员工数据池

作为 HR，我希望系统把多份 Excel 中同一个员工的数据合并到一行，便于检查和计算。

验收标准：

- 使用 `employee_name` 作为主要匹配键；
- 自动去除前后空格、全角空格；
- 同名员工生成 BLOCK 异常，提示检查 Excel；
- 每个字段值保存来源；
- 支持查看员工数据详情。

---

### US06 多次导入补充 Excel

作为 HR，我希望可以反复导入补充表，例如奖金表、扣款表、社保表。

验收标准：

- 每次导入生成独立 import batch；
- 新字段自动补齐；
- 冲突字段生成异常；
- 可查看字段来自哪次导入；
- 重新导入后需要重新检查。

---

### US07 数据检查

作为 HR，我希望系统在计算前自动发现明显错误，避免错发工资。

验收标准：

- 支持 BLOCK/WARN/INFO 三级异常；
- BLOCK 异常禁止计算；
- 异常列表支持按员工、字段、等级筛选；
- 异常详情显示原因、当前值、来源；
- 异常处理后可重新检查。

---

### US08 工资计算

作为 HR，我希望系统按公式自动计算工资，并保证结果可追溯。

验收标准：

- 无 BLOCK 异常时才能计算；
- 公式来自配置，不硬编码；
- 金额使用 Decimal；
- 计算结果精确到分；
- 每次计算生成版本；
- 保存每项工资计算过程。

---

### US09 工资解释

作为 HR，我希望点击某个员工后看到工资是怎么算出来的。

验收标准：

- 展示最终应发/实发工资；
- 展示每个工资项；
- 展示公式表达式；
- 展示参与字段值及来源；
- 展示 WARN 提醒；
- 支持复制解释文本。

---

### US10 导出 Excel

作为 HR，我希望导出长得像原工资主表的最终工资表，同时保留计算过程和异常报告。

验收标准：

- 复制工资主表作为模板导出新 Excel 文件；
- 原始 Excel 不被修改；
- 导出包含最终工资表，尽量保留原表样式、列宽、表头和格式；
- 可选包含计算过程、异常报告、来源明细；
- 支持重复下载历史导出文件。

## 4. 核心数据模型

### 4.1 Employee Data Pool Schema

员工数据池是当前 Salary Run 内的聚合视图。

```json
{
  "salary_run_id": "run_202607_001",
  "employee_ref_id": "emp_ref_001",
  "employee_name": "张三",
  "status": "NORMAL",
  "fields": {
    "base_salary": {
      "field_name": "基本工资",
      "value": "8000.00",
      "value_type": "money",
      "source": {
        "file_id": "file_001",
        "file_name": "7月工资主表.xlsx",
        "sheet_name": "工资表",
        "row_index": 12,
        "column_name": "基本薪资",
        "import_batch_id": "batch_001"
      },
      "manual_modified": false,
      "confidence": 0.98
    }
  }
}
```

### 4.2 标准字段类型

| 类型 | 说明 | 示例 |
|---|---|---|
| string | 文本 | 员工姓名、部门 |
| money | 金额，Decimal | 基本工资、奖金 |
| number | 普通数字 | 出勤天数 |
| percent | 比例 | 提成比例 |
| date | 日期 | 入职日期 |
| enum | 枚举 | 员工状态 |
| boolean | 布尔 | 是否计薪 |

## 5. 输入输出定义

### 5.1 输入

| 输入 | 格式 | 必填 | 说明 |
|---|---|---|---|
| 工资主工作簿 | `.xlsx` | 是 | 至少包含姓名和一个工资金额字段 |
| 补充 Excel | `.xlsx` | 否 | 可重复上传 |
| 字段配置 | YAML/JSON | 是 | 标准字段、别名、类型、必填 |
| Sheet 识别配置 | YAML/JSON | 是 | Sheet 类型、关键词、阈值 |
| 检查规则配置 | YAML/JSON | 是 | 检查项和异常等级 |
| 公式配置 | YAML/JSON | 是 | 工资项公式、依赖、舍入 |

### 5.2 输出

| 输出 | 格式 | 说明 |
|---|---|---|
| 员工数据池 | 数据库记录/API JSON | 聚合后的员工工资数据 |
| 异常报告 | 页面/API/Excel | 数据检查结果 |
| 工资计算结果 | 数据库记录/API JSON | 最终工资项 |
| 工资解释 | 页面/API/Excel | 公式、参数、来源、结果 |
| 最终工资表 | `.xlsx` | 基于工资主表模板新生成的导出文件 |

## 6. 异常场景

| 场景 | 等级 | 处理 |
|---|---|---|
| 文件无法解析 | BLOCK | 上传失败 |
| 无可识别 Sheet | BLOCK | 人工指定 Sheet |
| 缺少姓名列 | BLOCK | 人工映射或重新上传 |
| 姓名为空 | BLOCK | 修正或忽略该行 |
| 姓名重复 | BLOCK | 检查 Excel 后重新上传或修正数据 |
| 必填工资字段缺失 | BLOCK | 补录或导入补充表 |
| 金额格式非法 | BLOCK | 修正字段值 |
| 数据冲突 | BLOCK/WARN | 按字段配置决定 |
| 金额为负 | WARN/BLOCK | 按字段配置决定 |
| 工资异常偏高 | WARN | 提醒复核 |
| 导出失败 | BLOCK | 重试或查看系统错误 |

## 7. 非功能需求

| 类别 | 要求 |
|---|---|
| 性能 | 1 万行以内 Excel，解析和检查应在可接受时间内完成；具体指标以压测为准 |
| 安全 | 本机单用户登录；上传文件限制类型和大小；敏感日志脱敏 |
| 审计 | 上传、映射、异常处理、计算、导出均记录操作日志 |
| 可追溯 | 每个字段值、每个工资项都能追溯来源 |
| 可恢复 | 服务重启后任务状态、计算结果、导出记录不丢失 |
| 可测试 | 核心规则、公式、检查引擎必须有单元测试 |
| 可配置 | 字段、Sheet、检查、公式、导出列不硬编码 |

---

## 8. MVP 成功标准

1. HR 能独立完成一次从导入到导出的工资核算；
2. 系统能自动发现关键异常并阻断计算；
3. 工资计算结果可解释、可追溯；
4. 原始 Excel 不被修改；
5. 任务状态和历史结果可恢复；
6. 开发团队可以根据文档直接拆分接口、页面、引擎任务。
