# 01 - Project Scope（项目边界）

> Project: Smart Salary Engine（SSE）  
> Version: V1.2  
> Status: MVP 可开发基线  
> Last Updated: 2026-07-06

---

## 1. 项目定位

Smart Salary Engine（SSE）是一个围绕 **Excel 工资工作流** 设计的智能薪资数据整理与工资核算平台。

系统不替代 HR、OA、ERP、考勤、社保、个税或银行发薪系统，而是解决 HR 在工资核算前后最痛苦的部分：

- 多份 Excel 数据来源不统一；
- Sheet、列名、字段格式不稳定；
- 人工复制粘贴容易错；
- 补充数据反复导入，来源难追溯；
- 工资计算过程难解释；
- 发薪前异常难集中检查。

一句话定位：**SSE 是 HR 的 Excel 工资核算整理台，不是人事系统。**

---

## 2. 产品目标

### 2.1 第一阶段目标（MVP）

MVP 必须完成从 Excel 导入到导出工资表的闭环：

1. 导入工资主工作簿；
2. 导入补充 Excel，可重复导入；
3. 自动识别 Sheet 类型；
4. 自动识别列名与字段；
5. 按姓名建立员工数据池；
6. 发现重名员工并阻断计算，提示检查 Excel；
7. 自动补齐工资计算所需字段；
8. 自动检查缺失、重复、逻辑异常、金额异常；
9. 严重异常阻断工资计算；
10. 按配置公式计算工资；
11. 保存每个工资项的计算过程；
12. 生成工资解释；
13. 以工资主表为模板导出新的工资 Excel，不修改原始 Excel。

### 2.2 非目标

第一阶段不做：

- 员工档案长期维护；
- 考勤打卡；
- 请假审批；
- 社保申报；
- 个税申报；
- 个税计算；
- 银行发薪；
- OA/ERP 集成；
- 多公司组织架构管理；
- 复杂薪酬绩效审批流。

## 4. 产品边界

### 4.1 系统负责

- Sheet 自动识别；
- 列名自动识别；

## 5. 产品原则

| 编号 | 原则 | 落地要求 |
|---|---|---|
| P1 | Excel 是唯一业务数据来源 | 系统不凭空创建工资业务数据，只从用户上传文件提取 |
| P2 | 系统不维护员工资料 | 员工数据池只服务本次工资核算，不作为长期档案库 |
| P3 | 姓名是唯一匹配键 | 用户保证同一任务内无重名；发现重名直接 BLOCK，不自动合并或拆分 |
| P4 | 所有数据必须记录来源 | 字段值必须保存来源文件、Sheet、行、列、导入批次 |
| P5 | 所有工资必须能够解释 | 每个工资项保存公式、参数、来源、结果 |
| P6 | 所有计算必须保存全过程 | 不只保存最终工资，也保存中间项和计算 trace |
| P7 | 严重异常禁止工资计算 | BLOCK 级异常未处理前，计算按钮不可用 |
| P8 | 程序永远不修改原始 Excel | 导出时复制工资主表作为模板生成新文件，原始文件只读保存 |
| P9 | 工资规则配置驱动 | Sheet、字段、检查、公式、导出列尽量配置化 |
| P10 | 人工补录允许作为正式数据 | 人工补录必须记录处理人、时间、原因，并标记为人工来源 |

## 6. 核心业务对象

| 对象 | 含义 |
|---|---|
| Salary Run | 一次工资核算任务，例如“2026-07 工资核算” |
| Import Batch | 一次 Excel 导入批次，主表和补充表都算批次 |
| Workbook File | 上传的 Excel 文件元信息 |
| Sheet Mapping | 系统识别出的 Sheet 类型与置信度 |
| Column Mapping | Excel 原始列与标准字段的映射关系 |
| Employee Data Pool | 本次 Salary Run 内聚合后的员工数据池 |
| Field Value | 员工某个标准字段的值及其来源 |
| Check Issue | 数据检查产生的异常或提示 |
| Formula Config | 工资项公式配置 |
| Calculation Trace | 工资计算过程明细 |
| Export File | 导出的工资文件 |

---

## 7. 标准业务流程

1. HR 创建工资核算任务；
2. 上传工资主工作簿；
3. 系统解析 Excel，识别 Sheet 与列；
4. 系统建立员工数据池；
5. HR 上传补充 Excel；
6. 系统将补充字段合并到员工数据池；
7. 系统执行数据检查；
8. HR 处理缺失、冲突、异常；如发现重名则检查 Excel 后重新导入；
9. 系统确认无 BLOCK 异常；
10. HR 发起工资计算；
11. 系统生成最终工资、过程明细、解释；
12. HR 预览结果；
13. HR 导出 Excel。

## 9. 技术方案

### 9.1 Frontend

- Vue 3
- TypeScript
- Vite
- Element Plus
- Pinia
- Vue Router
- Axios

### 9.2 Backend

- Python >= 3.13, < 3.15
- FastAPI
- Uvicorn
- Pandas
- OpenPyXL
- Pydantic
- SQLAlchemy
- SQLite（MVP）/ PostgreSQL（后续生产）
- Decimal 金额计算

说明：Python 3.14 当前已经有正式维护版本，但工资系统更看重依赖兼容和稳定交付。MVP 建议使用 `>=3.13,<3.15` 的版本约束，实际锁定版本由依赖兼容测试决定。

### 9.3 存储方案

MVP 按本机单机使用设计，但不允许只靠内存或临时文件保存状态。最低要求：

- SQLite 保存任务、导入记录、映射、异常、计算结果、操作日志；
- 本地文件目录保存原始上传文件和导出文件；
- 原始 Excel 永久只读，不做覆盖写入；
- 导出文件由工资主表模板复制生成，尽量保留原表格式。

## 11. MVP 验收标准

MVP 交付时必须满足：

1. 能创建工资核算任务；
2. 能上传 `.xlsx` 文件；
3. 能识别主工资表、补充表、异常 Sheet；
4. 能识别姓名、基本工资、奖金、扣款等标准字段；
5. 能展示识别失败的 Sheet/列并允许人工修正；
6. 能按姓名建立员工数据池；
7. 能发现重名并生成 BLOCK 异常；
8. 能保存字段来源；
9. 能执行 BLOCK/WARN/INFO 三级检查；
10. BLOCK 异常未处理时不能计算工资；
11. 能按配置公式计算工资；
12. 金额计算使用 Decimal，最终结果精确到分；
13. 能展示每名员工每个工资项的计算过程；
14. 能以工资主表为模板导出最终工资 Excel；
15. 原始 Excel 文件内容不被修改；
16. 重启后任务、异常、计算结果仍可查询。
